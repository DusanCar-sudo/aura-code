# Aura Code is now a formally verified Abstract Agent Machine

Most AI coding agents are described in marketing language: "intelligent," "autonomous," "powerful." We wanted something more honest — a description that could be *checked*, not just believed.

So we built `:machina` — a tool that doesn't describe Aura's architecture from memory or from a README. It reads the actual source tree, makes specific claims about specific lines, and verifies every claim against the live code before publishing anything.

This week, we ran it against the full codebase. All 7 structural claims came back verified.

## What we found, verified

Aura is modelled as a **5-tuple Abstract Agent Machine**:

```
AAM = (S, P, O, δ, s₀)
```

- **S** — the state space: conversation history plus loop counters
- **P** — 27 finite, enumerable tool primitives (confirmed by direct count across `src/tools/*.ts`)
- **O** — the oracle: the only swappable part. Claude, GPT, Gemini, MiMo, or a fully local Ollama model
- **δ** — the transition function: consult the oracle, run its output through a safety gate, execute tools, fold results back into history
- **s₀** — empty history plus your task as the first message

The practical consequence: swap the oracle and the machine is **unchanged**. S, P, δ, and s₀ stay identical. That's not a design promise — it's a structural property `:machina` confirmed line-by-line against `src/agent/loop.ts`.

## The halting bound, made explicit

An *unbounded* version of this machine would be Turing-complete — which means the Halting Problem applies: no general test could say whether an arbitrary task ever finishes.

Real Aura isn't that machine, on purpose. `maxTurns: 150` (`src/config/defaults.ts:17`) and context-window compaction (`src/agent/compactor.ts`) are deliberate, finite bounds that buy guaranteed termination at a small cost in theoretical power. We'd rather have a system that always stops than one that's theoretically more powerful and occasionally doesn't.

## Why we're publishing this

Every claim in this post is either a direct quote from `:machina`'s verified output, or a number we re-checked by hand with a plain `grep` against the live source — including catching and correcting two small inaccuracies in our first draft (a tool count that needed re-verifying across files, and a compaction-policy description that had the primary rule and the fallback rule backwards).

That correction process is the point. We'd rather publish something smaller and true than something impressive and wrong.
