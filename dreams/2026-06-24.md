## Lessons
- [configuration] Path resolution for commands like `:viz` is sensitive to the working directory; if dashboards end up in wrong folders, fix relative paths rather than relying on `$HOME` assumptions.
- [safety] The `❌ Failed (1 turns) [SAFETY ON]` message means the task was blocked by the safety checker; toggling safety off via Telegram and re-sending the exact task can recover it, but confusion arises if the user forgets to re-submit after toggling.
- [multi-agent] Linking multiple Aura instances (desktop, phone, laptop) works best when you commit a shared setup document (e.g., `aura_desktop_setup.md`) that explains how to reach the same board HTML file and file paths, then each instance reads and executes it.
- [deployment] Static websites need an explicit `:approve all` to bypass safety when deploying; partial command variants (e.g., `:approve` without `all`) or typos like `:aprove all` will silently fail until the correct phrase is used.
- [tooling] A `HistoryMessage` union type burst across modules can break builds; fix it by searching for every usage of the affected type and syncing the new `tool_result` variant shape before iterating.
- [board-design] When a user supplies an HTML mockup and an audio recording, always transcribe the recording first, then reconcile the visual layout; this prevents implementing the wrong UI from guesswork.

## Patterns
- Unclear commands that circle until a precise, safety-bypassed `:approve all` triggers execution; often seen with multi-step creations like website deployment or local app setup.
- Multi-device orchestration: user asks Aura to create a manual for joining phone/laptop to the desktop, then follows that manual, occasionally requesting a summary of how the connection works for another AI (Claude or the phone Aura).
- Board integration cycle: provide HTML mockup → transcribe design intent → implement → discover missing functions → rewire to a specific file → ask for instructions to share the board across devices.
- Safety‑toggle loops: a task is rejected with safety ON, the user manually disables safety, then retries, sometimes repeating because the original message wasn’t resent correctly.
- Build‑fix micro‑cycles: a single TypeScript error spawns a quick diagnosis + patch sequence, often involving reading a markdown note at `/tmp/fix-*.md`.
